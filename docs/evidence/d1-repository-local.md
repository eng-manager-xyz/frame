# D1 aggregate repository conformance

Issue 12's provider-free evidence is generated by the pinned Wrangler
`4.111.0` local D1 runtime, not by treating Python SQLite as a substitute for
D1. Run it from a clean checkout with:

```sh
scripts/frame d1-conformance
```

The suite creates an isolated temporary D1 database, applies every ordered
migration, and loads two tenants plus 208 bounded synthetic video rows. It
starts the compiled Rust/Wasm Worker with pinned Wrangler on an ephemeral
`127.0.0.1` port, using a cryptographically random per-run token that is never
written to the repository or evidence. All repository behavior is invoked over
that HTTP boundary. CLI access is limited to migrations, controlled fixtures,
supplementary reference/D1 read-query parity, independent final-state
inspection, and `EXPLAIN QUERY PLAN`. Reference SQLite is supplementary parity
evidence for the checked-in SQL and trigger operation, not a substitute for the
runtime adapter.

The local report covers:

- found/not-found behavior for video, upload, media-job, worker-job, and
  organization aggregates;
- indistinguishable cross-tenant misses;
- 1- and 100-item keyset-page boundaries, including tied timestamps and a
  lookahead row without duplicates or omissions;
- 100-parameter query limits and 99-identifier chunking;
- a trigger-guarded D1 operation, invoked through `D1Database.batch`, that
  atomically reserves idempotency, applies the revision fence, inserts the
  outbox event, stores the response, suppresses its transient envelope with
  `RAISE(IGNORE)`, and exposes the actual `changes = 4` batch metadata for
  those retained trigger effects while leaving no envelope row;
- exact replay, same-key/different-payload conflict, stale and cross-tenant
  conflicts, injected mid-operation constraint rollback, and two concurrent
  HTTP requests that both complete with exactly one applied result;
- repository-boundary rejection for persisted media/worker attempts above
  `u32::MAX`, semantic page/bulk corruption, and padded write titles;
- active upload/media rows with a mismatched organization, including direct
  read hiding and exact snapshot counts for both affected tenants;
- an expired pre-dispatch adapter deadline (`deadline_ms=0` and
  `query_dispatched=false`); this does not claim in-flight cancellation;
- restored schema-fault mapping to `repository_unavailable` without SQL or
  provider error text;
- query plans tied to the checked-in SQL and actual Worker telemetry containing
  only query class, duration, row count, retry count, bookmark availability,
  and a fixed outcome code.

The machine-readable artifact is
`target/evidence/d1-repository-conformance.json`. CI retains it beside the
provider-free Worker bundle. The artifact records actual parsed Worker
telemetry; no sample is synthesized. Tenant identifiers, bindings, SQL text,
provider errors, row values, random tokens, and temporary paths are excluded
from telemetry and the report.

This evidence does not claim production replication lag, D1 session
bookmarks (the selected Workers binding does not expose them), provider
network timeouts, or remote capacity/contention. Those remain protected
provider gates rather than locally fabricated records.
